Floor Plan Understanding Challenge

Given a 2D floor plan image, automatically detect, classify, and localize structural elements - and produce a machine-readable building model.

🏗️ Background

Architectural floor plans are the primary communication medium in the construction industry. Billions of existing buildings have their layouts locked inside scanned drawings or raster images - inaccessible to modern BIM (Building Information Modeling) tools.

Automating the extraction of structured floor plan data from images is a high-value open problem in AEC (Architecture, Engineering, and Construction). This challenge asks you to build that bridge: from raw image to structured, semantic building model.

📁 The Dataset

You are provided with 500 training floor plans and 100 test floor plans derived from a real-world residential dataset. All samples are single-floor buildings - multi-floor entries have been excluded. Annotations (SVG) are provided for training only; test images have no annotations.

Images

  • Format: PNG, normalized resolution (~1000 x 1500 px)
  • Content: Rendered 2D floor plan, single floor per file
  • Coordinate space: Directly matches the SVG width × height - no rescaling needed

Annotations (SVG)

Each annotation is a standard Scalable Vector Graphics (SVG) file — an XML-based format where rooms, walls, doors, and windows are represented as <polygon> elements grouped inside <g> tags. The semantic meaning of each element is encoded in the CSS class attribute of its group (e.g. Space LivingRoom, Wall External). SVG units match pixel coordinates in the corresponding PNG — no rescaling is needed.

Parsing approaches (not limited to):

  • xml.etree.ElementTree — Python standard library, lightweight, sufficient for most cases
  • lxml.etree — faster, supports XPath queries for more complex traversal
  • BeautifulSoup with xml parser — convenient for exploratory work
  • Direct string / regex parsing of the points attribute if only coordinates are needed

Each SVG encodes the complete floor plan structure using semantic CSS class attributes. All geometry is represented as polygon coordinates in SVG units aligned to the image dimensions.

Reading Polygon Coordinates from SVG

Each room or wall is a <polygon> element inside a <g> group. Read the points attribute — it is a space-separated list of x,y pairs in SVG units (which equal pixel positions in the matching .png): 

<g class="Space LivingRoom">
  <polygon points="100.5,80.2 300.0,80.2 300.0,250.0 100.5,250.0" />
</g>
Note: Some <g> groups contain child <g> elements. Always check for a <polygon> at any depth within the group, and test if polygon_elem is not None: (not a boolean check) because an empty element evaluates to False in Python.

🎯 Tasks

All stages are mandatory and build on each other.

Stage 1

Semantic Room Segmentation

Given a floor plan image, produce a pixel-level segmentation assigning each spatial region to a room type class.

Stage 2

Room & Wall Outline Detection

Instead of coloring pixels, draw the exact shape of each room and wall as a polygon - a list of corner coordinates. Your output describes where each room is and what type it is (bedroom, kitchen, etc.).

Stage 3

IFC Generation

Convert your predicted floor plan model into a valid IFC file using ifcopenshell with correct spatial hierarchy. IFC coordinates must be in metres: multiply all pixel coordinates by 0.01 (1 pixel = 0.01 m). Use a default storey height of 3.0 m. Set each room as an IfcSpace entity with its Name property equal to the room type label (e.g. "LivingRoom", "Bedroom") so room types are carried into the BIM model.

Example output - raw image → Stage 1 (pixel labels) → Stage 2 (room outlines)
Important: The SVG annotations are provided for training only. At inference time your model must work from the image alone - do not pass SVG files through at test time.

ifc/ - IFC Space Name Strings

When creating each IfcSpace, set its Name to the canonical string from the table below. The evaluator accepts common aliases (e.g. Bathroom, WC, Living Room) but using the canonical names is safest. Walls, doors, and windows do not need to be IfcSpace entities.

2BedroomBedroom

Background (0), Wall (7), Door (8), and Window (9) are not represented as IfcSpace entities — only rooms (indices 1–6, 10) become spaces.

🛠️ What You Can Use

  • Any ML framework - PyTorch, TensorFlow, JAX, etc.
  • Pre-trained vision models - fine-tuning is allowed and encouraged
  • ifcopenshell for IFC generation (Stage 3)
  • opencv-python, Pillow, shapely, rasterio for geometry processing
  • Standard data augmentation libraries

Not allowed: Using the provided SVG annotations as direct pass-through at inference time. Training on external floor plan datasets is permitted with disclosure.
